## Context

The repository currently has a Linux-only CI workflow in `.github/workflows/ci.yml`: one job named `Linux CI` on `ubuntu-latest` that runs `mise format` and `mise run --force test`. The release workflow is cargo-dist generated and computes build matrices from cargo-dist configuration, while the current `release-automation` spec explicitly requires Linux-only artifacts and explicitly excludes non-Linux artifacts.

The macOS Seatbelt sandbox capability now has platform-specific code and integration tests. Those tests are `#[cfg(target_os = "macos")]` and skip only if `/usr/bin/sandbox-exec` is unavailable, so they need a macOS runner to provide meaningful continuous validation.

## Goals / Non-Goals

**Goals:**

- Add GitHub Actions macOS CI that runs on pull requests, pushes to `main`, and manual dispatch.
- Reuse the same validation boundary as Linux: `mise format` and `mise run --force test`.
- Update release automation requirements from Linux-only artifacts to Linux plus Apple Silicon macOS artifacts.
- Configure cargo-dist so the release plan includes `aarch64-apple-darwin` artifacts.
- Validate the release plan before declaring implementation complete.

**Non-Goals:**

- Do not add Windows CI or Windows release artifacts.
- Do not add Intel macOS (`x86_64-apple-darwin`) release artifacts.
- Do not replace cargo-dist release generation with a hand-written release implementation.
- Do not change release-cut versioning or changelog behavior.
- Do not add secret preflight checks; privileged publishing should continue to fail visibly through the underlying workflow when secrets are absent or under-scoped.

## Decisions

### Add a separate macOS CI job instead of making the existing job matrix-only

Keep the existing Linux job behavior intact and add a dedicated macOS job. This makes the workflow summary explicit and avoids accidentally weakening Linux validation while introducing macOS coverage.

Alternative considered: convert the current job to a `matrix.os` job. Rejected as the default because the existing spec names Linux validation explicitly and a separate job makes platform-specific failures easier to identify.

### Run the same mise tasks on macOS

The macOS CI job should install the mise-managed toolchain, add Rust formatting/linting components, run `mise format`, and run `mise run --force test`. This keeps local and CI validation aligned with repository guidance.

Alternative considered: run direct `cargo` commands on macOS. Rejected because the repository workflow requires mise tasks as the development and validation boundary.

### Let macOS Seatbelt tests execute normally on macOS runners

Do not add new environment gates or ignored tests. The existing tests already skip only when `/usr/bin/sandbox-exec` is unavailable. On GitHub-hosted macOS runners, the tests should run and fail normally if Seatbelt behavior regresses.

Alternative considered: add a CI-specific test filter. Rejected because it would weaken the exact macOS behavior the CI job is meant to validate.

### Configure cargo-dist targets rather than manually editing generated release behavior

Cargo-dist owns the generated release workflow. The implementation should update cargo-dist configuration in `dist-workspace.toml` and regenerate/validate the generated workflow if cargo-dist requires it. The final release plan must include Linux and Apple Silicon macOS artifacts.

Alternative considered: directly edit `.github/workflows/release.yml` to hard-code macOS jobs. Rejected because the file is generated by cargo-dist and manual changes risk being overwritten or diverging from cargo-dist expectations.

### Publish Apple Silicon macOS artifacts only

The release should publish macOS artifacts for `aarch64-apple-darwin` only. Intel macOS (`x86_64-apple-darwin`) is intentionally out of scope because those machines are no longer a relevant target for this project.

Alternative considered: publishing both Apple Silicon and Intel macOS artifacts. Rejected because it adds runner/build/release complexity for a platform variant the project does not intend to support.

## Risks / Trade-offs

- macOS CI can be slower or consume scarcer GitHub-hosted runner minutes → Keep the job focused on required mise validation only.
- Seatbelt availability or behavior can vary across macOS runner images → Let tests skip only on missing `/usr/bin/sandbox-exec`; otherwise treat failures as real regressions.
- Cargo-dist generated workflow may need regeneration after target config changes → Use cargo-dist plan/generation commands and review generated diffs instead of hand-editing generated sections blindly.
- Cross-target macOS publishing may require linker or host constraints → Prefer native macOS runners for the Apple Silicon artifact through cargo-dist's generated matrix.
- Homebrew formula publication may need artifact naming expectations checked after adding macOS archives → Validate the cargo-dist plan and generated upload files.

## Migration Plan

1. Add a macOS CI job to `.github/workflows/ci.yml` using the same checkout, mise, Rust component, format, and test steps as Linux.
2. Update cargo-dist configuration in `dist-workspace.toml` to include `aarch64-apple-darwin` alongside existing Linux targets.
3. Regenerate or validate `.github/workflows/release.yml` according to cargo-dist's expected workflow generation process.
4. Update `openspec/specs/release-automation/spec.md` through the delta spec sync flow after implementation.
5. Validate with `mise format`, `mise run --force test`, and a cargo-dist plan command that shows the Apple Silicon macOS artifact.
6. Roll back by removing the macOS CI job and macOS cargo-dist targets; Linux CI/release behavior remains the baseline.

## Open Questions

None. The target strategy is `aarch64-apple-darwin` only for macOS release artifacts.
